## 1. Архитектура (для разработчика)

- `app.js` — обёртка `track(goal, params)`: try/catch + проверка `typeof ym` (адблок у геймеров — норма; аналитика не должна ронять сайт). Полный реестр goal-id — в комментарии рядом с функцией.
- `tutorial.js` — собственная копия обёртки `tTrack()`: модуль по контракту не зависит от app.js.
- Поиск: второй дебаунс 1400 мс поверх рендерного (150 мс) — трекаются «устоявшиеся» запросы, не клавиши; `q` обрезается до 100 симв.
- `reagent_open` шлётся только при `pushHistory=true` (программные ре-рендеры back-nav/fork-switch не двоят счёт).
- Кеш-бастинг: `app.js?v=5`, `tutorial.js?v=2`.
- `scripts/create_metrika_goals.py` — идемпотентный API-создатель целей. ⚠ Токену нужны ОБА скоупа `metrika:read` И `metrika:write` (write не даёт read; проверено 2026-07-12, write-only токен → 403 на чтении). Владелец предпочёл создавать цели вручную.

## 2. Чек-лист целей (владелец создаёт вручную)

Метрика → Цели → Добавить → тип **«JavaScript-событие»** → идентификатор посимвольно. **Пока цель не создана — события отбрасываются без возможности восстановления**, поэтому создание целей = блокер сбора данных.

| Группа | Идентификаторы | Отвечают на вопрос |
|---|---|---|
| Вкладки | `tab_reactions` `tab_calculator` `tab_trees` `tab_graph` `tab_botany` `tab_stats` `tab_antag` | какие вкладки живые (tab_reagents не трекается — стартовый экран) |
| Контент | `reagent_open` `fork_select` | что смотрят, какие форки выбирают |
| Поиск | `search_used` `search_zero` | что ищут / **что не находят** (идеи контента!) |
| Фичи | `calc_run` `batch_plan` `reverse_used` `tree_built` `share_click` `antag_on` `strategy_to_batch` | пользуются ли калькуляторами/шарингом/антагом |
| Онбординг | `tutorial_start` `tutorial_done` `tutorial_skip` | воронка туториала, шаг отвала (`step` в params) |

## 3. Параметры событий (детализация)

Params пишутся независимо от целей и видны в **Отчёты → Содержание → Параметры визитов**:

- `reagent_open {reagent, tab}` — топ открываемых реагентов
- `fork_select {fork}` — рейтинг форков по реальному выбору (⇒ приоритет поддержки форков)
- `search_used/zero {q, tab, results}` — тексты запросов; `search_zero` = прямой бэклог («что игроки ожидали найти»)
- `calc_run {target, amount}`, `batch_plan {targets}`, `tree_built {reagent}`, `reverse_used {ingredient}`
- `share_click {tab, antag}` — что шарят (виральные поверхности)
- `strategy_to_batch {strategy}` — вовлечённость в antag-фичу
- `tutorial_start {auto}` (1=автозапуск новичку, 0=кнопка «?»), `tutorial_skip {step}`

## 4. Как читать через 2–3 недели

1. **Отчёты → Конверсии** — сравнение вкладок/фич между собой; сегментировать по источнику (гипотеза: Steam-трафик конвертится в калькулятор лучше прямых).
2. **Параметры визитов** — топы реагентов/форков/запросов; `search_zero` разобрать руками.
3. Воронка `tutorial_start → done/skip` + шаг отвала → правки онбординга.
4. Решения по roadmap: см. приоритеты в [SUMMARY.md](SUMMARY.md) §2 — пункты 4–6 ранжировать данными.
