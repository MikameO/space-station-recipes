# Дизайн: поиск предметов по картам станций (таб Maps)

- **Дата:** 2026-07-12
- **Статус:** одобрено, готов к реализации
- **Затрагивает:** новые `ss14_map_extractor.py`, `maps.js`, `maps/` (генерируемые данные); правки `index.html` (таб, скрипт), `config.py` (map-настройки форков), точечно `app.js` (регистрация таба) и `style.css`

## Проблема

Игроку нужно отвечать на вопрос «где на этой карте взять предмет X» — лом,
плазму, дефибриллятор, бикер. Раньше у автора был отдельный текстовый
инструмент (выбор карты → выбор ресурса → список мест); код не сохранился.
Воссоздаём внутри сайта, но с визуальным интерфейсом: интерактивная схема
станции с маркерами находок.

Ключевой факт: «где взять предмет» в SS14 — трёхслойная задача. Предмет
лежит на полу (entity на карте), внутри шкафа/ящика (`StorageFill` в
прототипе контейнера) или продаётся в вендомате (`VendingMachineInventory`).
Инструменты, видящие только первый слой, бесполезны: большинство предметов —
в контейнерах.

## Утверждённые решения (брейнсторминг 2026-07-12)

| Вопрос | Решение |
|---|---|
| Размещение | Новый таб «Maps» в существующем сайте |
| Интерфейс | Карта-центричный: canvas-схема + боковой список локаций |
| Глубина рендера | Пол (тайлы) + стены + двери; полный спрайтовый рендер отвергнут |
| Скоуп предметов | Все (фильтр = `Item`-компонент в наследовании прототипа), без ручного курирования |
| Форки | Все 18 сразу; отказоустойчивость обязательна |
| Данные | Per-map lazy JSON+PNG; `data.json` не трогаем |

## Данные и пайплайн

Новый скрипт **`ss14_map_extractor.py`** — отдельный от химического (свой
ритм regen), питается тем же `FORK_REGISTRY` из `config.py`, реиспользует
fetch-кэш.

**Дискавери файлов:** GitHub `git/trees?recursive=1` — один запрос на форк,
кэшируется. Из дерева берём `Resources/Maps/**` и `Resources/Prototypes/**`
(+ кастомные директории форков из `FORK_REGISTRY`).

**Фильтр станций:** исключаются поддиректории (`Dungeon/`, `Ruins/`,
`Shuttles/`, `Misc/`, `Salvage/`, `Test/`), карты с `meta.category != Map` и
карты, чей крупнейший грид меньше порога по числу сущностей (порог
подбирается эмпирически на инкременте E3); спорные случаи — ручной
allow/block-список в `config.py`.

**Реестр прототипов** (один раз на форк, с учётом наследования `parent`):

- id → имя, описание — для поиска;
- «предметность»: наличие `Item`-компонента в цепочке наследования —
  автоматический фильтр структурного мусора (стены, трубы, провода, декали);
- станционарное оборудование (вендоматы, диспенсеры, крио и пр.) — отдельная
  категория `machinery`, тоже ищется;
- `StorageFill.contents` шкафов/ящиков (записи с `prob < 1` сохраняют
  вероятность);
- инвентари вендоматов (`VendingMachineInventoryPrototype`);
- маяки: прототипы `DefaultStationBeacon*` — имя локации из прототипа
  (+ переопределение `NavMapBeacon.text` на карте, если задано).

**Парс карты (format 7).** Из YAML: `tilemap` (id → имя тайла), base64-чанки
тайлов, entity-группы `proto → [uid, pos, parent]`. Выход — два файла на карту:

| Файл | Содержимое | Оценка (gzip) |
|---|---|---|
| `maps/<fork>/<map>.png` | Подложка: пол + стены + двери, 1 px = 1 тайл. Цвет пола = средний цвет реальной текстуры тайла (спрайты уже харвестятся) | 20–80 KB |
| `maps/<fork>/<map>.json` | `bounds`; маяки `[имя, x, y]`; предметы `{proto: {name, positions[]}}`, позиция несёт способ получения: пол / шкаф (имя) / вендомат (имя) и `prob`, если < 1 | 30–60 KB |

Контейнеры и вендоматы разрешаются в экстракторе: фронт получает готовые
позиции и не знает о прототипах шкафов.

**Мультигрид:** файл карты содержит несколько гридов — независимых «плит»
тайлов (станция + мелкие структуры рядом: спутник ИИ, аванпосты; на Bagel —
грид «Syndi Puddle CS108», 326 сущностей против 25 039 у станции). PNG и
маркеры — только самый большой грид (станция). Предметы с прочих гридов в
индекс входят и в списке локаций помечаются именем грида из `MetaData`
(«на структуре Syndi Puddle CS108»), без маркера.

**Индекс** `maps/index.json`: форки → карты (id, имя, размеры, счётчики
предметов/маяков). Грузится при открытии таба; сами карты — лениво по клику.

## Фронтенд

Отдельный **`maps.js`** (прецедент — `tutorial.js`); в `index.html` — кнопка
таба, контейнер, `<script src="maps.js?v=1">`; в `app.js` — только
регистрация таба, если переключалка не generic. Меньше пересечений с
параллельными сессиями, правящими `app.js`.

**Лэйаут** (утверждённый мокап B): селектор карты (фильтруется активным
форком сайта) + поиск с автодополнением по предметам выбранной карты; слева
canvas, справа список локаций; на мобильном — стек.

**Canvas, три слоя:** PNG-подложка (`drawImage` + `image-rendering:
pixelated`) → подписи маяков (циан; скрываются на дальнем зуме) → маркеры
(зелёный = на полу, янтарный = в шкафу, фиолетовый = вендомат; легенда под
картой). Зум колесом и кнопками `+/−`, пан драгом (pointer events — работает
на таче). Перевод координат: `px = x − minX`, `py = maxY − y` (ось Y
инвертирована), из `bounds`.

**Поток:** таб → `index.json` → выбор карты → лениво PNG+JSON → ввод
предмета → маркеры + список локаций, сгруппированный по ближайшему маяку
(евклидово расстояние); клик по строке — зум к маркерам этой локации. Карта
без маяков → группа «без локации», голые координаты.

**Deep-links:** share-URL дополняется `map=<fork>/<id>&item=<proto>` через
существующий механизм. **Метрика:** goal-события «таб открыт», «карта
выбрана», «поиск выполнен» (прецедент `62c964c`).

## Обработка ошибок

- Экстрактор: карта не распарсилась (иная версия формата) → warning + скип
  карты; форк без карт → скип форка; regen не падает целиком. Счётчики
  «карт обработано / пропущено» — в вывод (M1-дисциплина).
- Фронт: сеть/404 по карте → плашка с retry, таб живёт; предмет не найден →
  честное «на этой карте не найден».

## Верификация (DoD)

- Самопроверка экстрактора (`assert`-блок на Bagel): Crowbar найден и его
  позиция совпадает с YAML, маяков > 20, размер PNG = `bounds`.
- Спот-чек вручную: 3 предмета (пол / шкаф / вендомат) против YAML.
- Фронт: e2e в preview кликами (выбор карты → поиск → маркеры → зум),
  скриншот; cache-bust `?v=N`; Метрика-событие видно в отладке.

## Не-цели (YAGNI)

- Полный спрайтовый рендер карты — отвергнут навсегда (мешает маркерам).
- Спавнеры случайного лута (`RandomSpawner`/`EntityTableSpawner`) — не в v1;
  слой добавится отдельным инкрементом (данные это позволяют).
- RU-имена в поиске (Fluent-локали форков) — не в v1; отдельный инкремент.
- Глобальный поиск «на каких картах есть X» — v2, потребует общий индекс.
- Процедурная генерация (руда астероидов, salvage-обломки) — вне скоупа:
  карта знает только статику.
- Рендер второстепенных гридов карты (мини-карты-инсеты для спутника ИИ,
  аванпостов и т.п.) — их предметы видны в списке с именем грида, но точек
  на карте нет; инсеты — кандидат в инкремент, если Метрика покажет спрос.

## План реализации (высокоуровнево)

1. **E1 — данные, один форк:** реестр прототипов vanilla + парс Bagel
   end-to-end (PNG + JSON + самопроверка). Риск формата снимается здесь.
2. **E2 — фронт MVP:** таб, селектор карт vanilla, поиск, canvas с
   подложкой/маяками/маркерами, список локаций, зум/пан. Верификация в preview.
3. **E3 — все карты vanilla:** фильтр станций, `index.json`, ленивая загрузка.
4. **E4 — 18 форков:** раскатка, отказоустойчивость, warnings; deep-links;
   Метрика-события; обновление README/CHANGELOG.
5. **E5+ (бэклог):** спавнеры, RU-поиск, «на каких картах есть X», тайлы
   второстепенных гридов.
